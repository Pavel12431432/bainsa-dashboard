# BAINSA Dashboard — Codebase Reference

Read this file instead of exploring the codebase. It is complete and up-to-date as of 2026-03-27.

## What this is

Instagram Story approval dashboard for BAINSA. Displays stories generated by the "Sofia" AI agent as 9:16 card previews. Users can edit stories (manually or via inline Sofia chat), approve/reject, browse by date, view version history, and view agent output in a slide-out drawer.

Live at `dashboard.pavelj.com` (Cloudflare tunnel -> localhost:3000). Systemd service: `bainsa-dashboard`.

## Tech stack

- **Next.js 16** (App Router) + TypeScript + React 19
- **Tailwind CSS 4** with `@theme inline` design tokens in `globals.css`
- **No database** — filesystem only (markdown files + JSON sidecars)
- **Auth**: single shared password, httpOnly cookie, `proxy.ts` guard
- **Font**: Space Grotesk (placeholder for Alliance No.1/No.2)
- **Zero external dependencies** beyond Next.js + Tailwind

## Important gotchas

- **SECURE_COOKIES=true** -> auth cookie has `Secure` flag -> `macmini:3000` (HTTP) won't work. Always use `dashboard.pavelj.com` (HTTPS).
- **Sandy Bridge CPU** -> no Turbopack, production build only (`npm run build`), `npm run dev` is painfully slow.
- **Server Actions** must redirect to final URLs (no double-redirect).
- **Rebuild required** after `.env.local` changes.
- **`export const dynamic = 'force-dynamic'`** needed on pages using `new Date()` to prevent static caching at build time.
- **No Co-Authored-By** in commit messages — user explicitly doesn't want this.
- **OpenClaw HTTP API doesn't work** on version 2026.3.23 — use `docker exec openclaw openclaw agent` CLI instead. See `lib/openclaw.ts`.
- **Card scaling**: StoryCard uses `useLayoutEffect` + `ResizeObserver` to scale cards. Do NOT try CSS-only approaches — `scale(calc(cqw))` produces a length not a number, and `tan(atan2(cqw, px))` breaks on mobile browsers. The current JS approach runs before paint (no CLS).

| `OPENCLAW_PASSWORD` | OpenClaw gateway password for Sofia chat API |

## Data flow

1. **Sofia** (OpenClaw agent) writes `YYYY-MM-DD.md` to `STORIES_PATH`
2. **Marco** (OpenClaw agent) writes `YYYY-MM-DD.md` to `MARCO_HANDOFFS_PATH`
3. Dashboard reads these files, parses stories, renders as cards
4. Edits write back to the same `.md` file via `replaceStory()`
5. Approvals stored in `YYYY-MM-DD.approved.json` sidecar next to stories
6. Version history stored in `YYYY-MM-DD.history.json` sidecar next to stories

## Story markdown format (Sofia output)

```markdown
---
## Story 1: Title Here

**Division:** Analysis
**Accent color:** #fe6203
**Layout:** headline-top-body-source

**Headline:** The punchy headline text

**Body:** Body text up to ~240 chars

**Source tag:** SourceName

**Corner accent:** >
```

Sections separated by `\n---\n`. Parser also supports embedded JSON blocks (future format).

## Data types (`types/index.ts`)

```typescript
interface Story {
  index: number;           // 1-based
  title: string;           // from "## Story N: Title"
  division: "Analysis" | "Projects" | "Culture" | string;
  accentColor: string;     // hex
  layoutTemplate: string;
  headline: string;
  body: string;
  sourceTag: string;
  cornerAccent: ">" | "+";
}

interface ApprovalState {
  approved: number[];   // 1-based story indices
  rejected: number[];
}

interface ComplianceResult {
  colorValid: boolean;    // accentColor matches division canonical
  headlineOk: boolean;    // <= 60 chars
  bodyOk: boolean;        // <= 180 chars
  sourcePresent: boolean;
  pass: boolean;
}

interface HistoryEntry {
  story: Story;
  label: string;
  timestamp: string;
}

const ACCENT_COLORS = { Analysis: "#fe6203", Projects: "#fe43a7", Culture: "#fe43a7" };
```

## Responsive behavior

- **Desktop (>=1024px)**: 4-column grid, cards auto-scale to fill column width (capped at 405px)
- **Tablet (<1024px)**: auto-fill columns, min 280px each
- **Mobile (<640px)**: single column, header uses CSS grid (2 rows), editor stacks vertically with FIELDS/SOFIA tab switcher (preview hidden), drawer full-width

## Auth flow

1. User visits any page -> `proxy.ts` checks `bainsa-auth` cookie
2. No valid cookie -> redirect to `/login`
3. Submit password -> Server Action compares against `DASHBOARD_PASSWORD` env var
4. Match -> sets `bainsa-auth` cookie (HMAC token, 7-day expiry) -> redirect to today's stories
5. Mismatch -> redirect to `/login?error=1`
6. Rate limit: 5 failed attempts -> 15min lockout (in-memory)

## API security

All mutation endpoints (`update`, `approve`, `chat`, `history POST`) require `X-Requested-With: fetch` header to prevent CSRF. All routes accepting date params validate with `isValidDate()` regex to prevent path traversal.

## Sofia inline chat

The story editor has an integrated chat panel for refining stories via natural language. Architecture:

- **Session persistence**: OpenClaw sessions keyed by `dashboard:edit:{date}:{index}:{uuid}`. UUID stored in localStorage (`sofia-session:{date}:{index}`). Reset button generates new UUID. OpenClaw manages conversation history server-side — client only sends the current message + story state, not full history.
- **API flow**: Client -> `POST /api/stories/[date]/[index]/chat` -> `docker exec openclaw openclaw agent --agent story-generator --session-id <id> --message <text> --json` via `child_process.execFile`.
- **Response parsing**: Sofia responds with explanation + optional JSON code block. `parseResponse()` extracts the JSON and applies field updates to the draft.
- **Accept/Revert**: Sofia's suggestions preview in fields with an inline diff banner (strikethrough old → green new). User explicitly accepts (auto-saves + records history) or reverts. This avoids ambiguity about what's saved.
- **Layout**: Desktop = preview | form | chat (3 columns). Mobile = FIELDS/SOFIA tab switcher.

## Version history

Each story can have a version timeline showing how it changed over time:

- **Storage**: `YYYY-MM-DD.history.json` sidecars, keyed by story index, capped at 50 entries per story.
- **Labels**: "Original" (auto-recorded on first edit), "Sofia: headline, body" (on accept), "Manual edit" (on save), "Restored: previous version" (on restore).
- **UI**: HISTORY(N) button toggles timeline. Click entry to preview (fields become read-only). RESTORE saves and records. BACK returns to current.
- **Original snapshot**: Uses an immutable `originalStory` ref set on mount — not `saved.current`, which gets mutated by `saveStory()` before `recordHistory()` runs.
