## Demo mode (for teammates / local dev)

Dashboard can run standalone with no OpenClaw/Docker. `.env.example` ships defaults that point at `samples/` (3 days of committed stories, variants, handoffs, instructions) and set `DEMO_MODE=true`. In demo mode, `chatWithAgent()` in `lib/openclaw.ts` short-circuits with canned responses:
- Variants generation -> 3 hardcoded template variants (text/bullets/quote)
- Inline story edit -> brief message + tiny JSON headline patch
- Regenerate/reconcile -> bumps `STORIES_PATH/{date}.md` mtime so the stale banner clears
- Free-form drawer chat -> echo-style demo message

Teammates: `cp .env.example .env.local && npm install && npm run build && npm start`. Runtime mutations (approvals, story history, ADAPTIVE history) in `samples/` are gitignored; the seed markdown + variants sidecars are committed.

## What this is

Instagram Story approval dashboard for BAINSA (Bocconi AI & Neuroscience Student Association). Displays stories generated by the "Sofia" AI agent as 9:16 card previews. Users can edit stories (manually or via inline Sofia chat), approve/reject with feedback, browse by date, search across all dates, view version history, export as PNG, chat with Marco/Sofia agents in slide-out drawers, explore A/B variants of each story via EDITOR/EXPLORE toggle, and manage Sofia's adaptive instructions via the Teach Sofia page. The Teach page also hosts Lorenzo, an editor agent that reviews human feedback on Sofia's stories and proposes targeted updates to ADAPTIVE.md. This project is for a competition against another team with the same assignment.

Live at `dashboard.pavelj.com` (Cloudflare tunnel -> localhost:3000). Systemd service: `bainsa-dashboard`.

## Important gotchas

- **No sudo** â€” Claude cannot run `systemctl` or any privileged commands. Tell the user to run them.
- **No Co-Authored-By** in commit messages â€” user explicitly doesn't want this.
- **SECURE_COOKIES=true** -> auth cookie has `Secure` flag -> `macmini:3000` (HTTP) won't work. Always use `dashboard.pavelj.com` (HTTPS).
- **Sandy Bridge CPU** -> no Turbopack, production build only (`npm run build`), `npm run dev` is painfully slow.
- **Rebuild required** after `.env.local` changes.
- **`export const dynamic = 'force-dynamic'`** needed on pages using `new Date()` to prevent static caching at build time.
- **Card scaling**: StoryCard uses `useLayoutEffect` + `ResizeObserver` to scale cards. Do NOT try CSS-only approaches â€” the current JS approach runs before paint (no CLS).
- **Sofia format**: Her SOUL.md must use concrete examples, not template hints like `(max 8 words)` â€” Sofia copies hints literally into output, breaking the parser.
- **Story markdown format changed**: Sofia now writes `## Story N` (no colon, no title). Both `parseStories.ts` and `serializeStories.ts` must match this format. The old format was `## Story N: Title`.
- **Field regex**: `extractField()` in `parseStories.ts` uses `[ \t]*` (not `\s*`) after the colon to avoid matching across newlines when a field value is empty.
- **Mobile keyboard + fixed panels**: `h-dvh`, `interactive-widget`, and `visualViewport` API all failed to fix iOS keyboard pushing slide panels. Left as a known issue.
- **html2canvas-pro can't render CSS `mask-image`** (also fails on inline `<svg>` `<mask>` and `<feColorMatrix>` filter â€” all leave the underlying box solid in export). Accent PNGs are tinted offscreen via canvas (`source-in`) into a data URL by `lib/tintedAccent.ts` and rendered as a plain `<img>` by `<TintedAccent>` in `StoryContent.tsx`. `exportCards.ts` preloads tinted variants before `html2canvas` runs so capture sees ready images.
- **Next.js Router Cache vs `force-dynamic`**: `force-dynamic` only stops *build-time* static caching. Next.js still caches RSC payloads client-side for ~30s during navigation. If a server file changed while the user was on another date, navigating back shows the *cached* RSC (stale) â€” `force-dynamic` does not help. Fix per-page: set a "data changed" flag in a tracker, and on mount refetch via a client-side `fetch(/api/...)` which bypasses the Router Cache. This is what `regenerateTracker` + the StoryGrid mount effect does.

### Lorenzo â€” editor agent flow

Lorenzo is invoked on-demand from the Teach Sofia page ("PROPOSE UPDATES" button). Not on a cron â€” a human triggers each review.

1. User clicks PROPOSE UPDATES â†’ `FeedbackInspector` modal shows last 7/14/30 days of signals (rejections, edits, variant activity, approvals) with expandable per-item details, plus an optional `Focus` textarea above the GENERATE button. A fourth "None" window option skips feedback entirely â€” Lorenzo then sees only the current ADAPTIVE.md and the focus, useful when the operator wants a precise change without dragging in unrelated signals. With None selected, focus becomes required and GENERATE stays disabled until it has content.
2. User optionally types an operator focus (max 500 chars) and clicks GENERATE PROPOSAL â†’ dashboard calls `POST /api/teach/propose` with `{ days, focus? }`
3. Route calls `collectFeedback()` (`lib/editorFeedback.ts`) which assembles a `FeedbackBundle` from story markdown, approval sidecars, history sidecars, and variants sidecars, plus the current ADAPTIVE.md and its last 3 versions
4. `runEditorAgent()` (`lib/editorAgent.ts`) wraps the bundle in a `<FeedbackBundle>` tagged block (and an `<OperatorFocus>` block when focus is present) and calls `chatWithAgent("lorenzo", sessionId, prompt)` â€” Lorenzo responds with a single JSON object (`status`, `summary`, `rationale[]`, `conflicts[]`, `proposedContent`). In operator focus mode (per his SOUL.md) he must return `status: "proposal"` and may use empty `signalRefs: []` for purely focus-driven rationale entries.
5. Route validates the response and writes it to `ADAPTIVE.proposal.json` sidecar (one pending proposal at a time)
6. UI shows the proposal in `ProposalView`: summary, conflict callout (amber), rationale bullets with clickable signal citations, and a diff vs current ADAPTIVE.md
7. Server computes warnings (`lib/proposalWarnings.ts`): `significant-shrinkage` when proposedContent is < 50% the length of the base, `unchanged-from-previous` (refine only) when the refine returned identical content. Surfaced as amber pills in `ProposalView`.
8. User clicks ACCEPT â†’ `/api/teach/propose/accept` writes proposedContent to ADAPTIVE.md and records the previous content in `ADAPTIVE.history.json` tagged `source: "editor-agent"` (surfaces as a small "lorenzo" tag in the HistoryTimeline)
9. User clicks REJECT (or DISMISS for no-changes) â†’ deletes sidecar
10. User clicks EDIT FIRST â†’ deletes sidecar, pre-fills the textarea with proposedContent, and sets a one-shot `pendingSaveLabel` so the user's next save is tagged "Edited Lorenzo proposal" in history
11. User clicks REFINE â†’ inline textarea â†’ nudge â†’ `POST /api/teach/propose/refine` reads the pending sidecar as `previousProposal`, re-collects feedback, and re-runs Lorenzo with `<PreviousProposal>` + `<Nudge>` blocks appended to the prompt. New sidecar carries `refineHistory` (breadcrumb) and a one-level `previousProposal` for undo.
12. User clicks UNDO (next to breadcrumb) â†’ `POST /api/teach/propose/undo-refine` restores the proposal's `previousProposal` as the new sidecar state (one level only).

**Testing**:
- `npm run editor-agent:test` runs 5 hand-crafted fixtures (too-corporate, conflicting-signals, weak-signal, stale-rule, happy-path) against the real Lorenzo via OpenClaw CLI and dumps reports to `scripts/editor-agent-output/`. After any SOUL.md tweak, run it and check `git diff scripts/editor-agent-output/`.
- `npm run stub-proposal <scenario>` writes a canned proposal sidecar for fast UI iteration without calling Lorenzo. Scenarios: `proposal` / `conflict` / `no-changes` / `stale` / `shrinkage` / `refined`. REJECT/DISMISS in the UI clears it.

**Lorenzo's behavior rules** (enforced in his SOUL.md): cluster feedback into themes before deriving rules; never build a rule from a single signal; preserve existing ADAPTIVE.md verbatim unless directly contradicted (hard floor: `proposedContent` >= 80% of `currentAdaptive` length); flag conflicts instead of silently picking sides; return `status: "no-changes"` when signal is too weak; on refine, always acknowledge the nudge in the summary's first sentence â€” never a token cosmetic edit. **Operator focus mode** (when prompt contains `<OperatorFocus>`): must return `status: "proposal"` (no-changes is forbidden); treat focus as the highest-priority instruction; signal arrays may be empty (None mode) â€” work purely from the focus and `currentAdaptive`; use `signalRefs: []` for purely focus-driven rationale entries (the only case where empty refs are allowed) and make the focus-driven origin clear in the bullet text; raise a `conflicts` entry when the focus contradicts an existing rule or strong pattern; acknowledge the focus in the summary's first sentence. The 80% floor still applies.

## Instagram direct publishing

ExportDialog has a **DESTINATION** toggle: DOWNLOAD (existing PNG/ZIP flow) or INSTAGRAM (posts as Stories via Graph API). Posts run sequentially with no delay â€” the dialog must stay open until done. A background queue with delay/scheduling is a future feature.

Flow per story:
1. Client renders the card via `captureStoryToBlob(story, "image/jpeg", 8/3)` â€” JPEG at 1080Ã—1920 (Meta's Stories max; render at higher resolution gets rejected with the cryptic `code=1` "unknown error").
2. Client POSTs the JPEG bytes to `/api/instagram/publish` (Content-Type: image/jpeg, X-Requested-With: fetch).
3. Server saves bytes to `$TMPDIR/bainsa-ig-temp/<32hex>.jpg` via `lib/igTempStore.ts` and computes the public URL `${IG_PUBLIC_BASE}/ig-temp/<key>`. The `app/ig-temp/[key]/route.ts` handler serves GET/HEAD with `Content-Length` + `Cache-Control: public, max-age=600`. Path is whitelisted in `proxy.ts` so it bypasses auth.
4. Server calls Meta: `POST /{IG_USER_ID}/media?media_type=STORIES&image_url=<public-url>` â†’ polls container `status_code` until FINISHED â†’ `POST /{IG_USER_ID}/media_publish?creation_id=â€¦`.
5. On success, schedules `deleteImage(key)` after 10 minutes (Meta has fetched it by then). On error, returns 502 with the Meta message (`metaCall` logs the full Meta error JSON to stdout for debugging â€” `error_user_msg` is often hidden inside the wrapper `message`) and the temp file is deleted immediately.

On success, ExportDialog calls `onSuccess(message)` on the parent (StoryGrid) which shows a bottom-center toast that auto-dismisses after 4s. Progress bar trickles smoothly during each request (NProgress-style â€” caps at 90% until the request actually returns).

`/api/instagram/check` is a diagnostic GET endpoint that validates env vars, token, IG user lookup, Pageâ†’IG link, and granted permissions. Hit it in a browser when posting breaks.

**Why self-host through the tunnel?** We tried several intermediaries before this. ImgBB returns 403 to Meta's crawler IPs. Catbox went down with "uploads paused". Cloudflare R2's S3 endpoint refused TLS handshakes from this network. Cloudflare Workers on `*.workers.dev` and R2 buckets on `*.r2.dev` are blocked for Meta scrapers at the platform level (and adding a custom domain on a Worker still got 9004 from `/media` even when the FB debugger scraped fine â€” IG's fetcher seems pickier than the debugger). Self-hosting via the existing Cloudflare tunnel at `dashboard.pavelj.com` works because it's a normal origin and Meta is allowlisted as a verified bot there.

## Data flow

1. **Marco** writes `YYYY-MM-DD.md` to `MARCO_HANDOFFS_PATH` (daily at 12:00)
2. **Sofia** reads Marco's handoff + brand guidelines, writes `YYYY-MM-DD.md` to `STORIES_PATH` (daily at 12:30)
3. Dashboard reads these files, parses stories, renders as 9:16 cards
4. Edits write back to the same `.md` file via `replaceStory()` in `serializeStories.ts`
5. Approvals stored in `YYYY-MM-DD.approved.json` sidecar next to stories (includes optional `feedback` map: story index -> rejection reason)
6. Version history stored in `YYYY-MM-DD.history.json` sidecar next to stories
7. Human triggers Lorenzo from Teach page -> pending proposal stored in `ADAPTIVE.proposal.json` in `SOFIA_INSTRUCTIONS_PATH` -> on accept, proposal content replaces ADAPTIVE.md and previous version goes to `ADAPTIVE.history.json` tagged `source: "editor-agent"`

## Story markdown format (current Sofia output)

```markdown
---
## Story 1

**Division:** Culture
**Accent color:** #fe43a7
**Layout:** top
**Content type:** text
**Headline size:** default
**Body weight:** regular
**Text align:** left
**Corner size:** small
**Accent bar:** bottom

**Headline:** AI funding keeps breaking records

Sections separated by `\n---\n`. Parser also supports embedded JSON blocks (future format). Note: no colon or title after `## Story N`.

## Data types (`types/index.ts`)

```typescript
type Layout = "top" | "center" | "bottom";
type ContentType = "text" | "bullets" | "quote";
type HeadlineSize = "large" | "default" | "compact";
type BodyWeight = "regular" | "semibold";
type TextAlign = "left" | "justify";
type CornerSize = "small" | "medium";
type AccentBar = "bottom" | "top" | "none";
type GhostAccent = "none" | "bottom-right" | "center" | "top-left";

interface Story {
  index: number;           // 1-based
  title: string;           // legacy, usually empty
  division: "Analysis" | "Projects" | "Culture" | string;
  accentColor: string;     // hex e.g. "#2c40e8"
  headline: string;        // max 80 chars
  body: string;            // max varies by contentType
  sourceTag: string;
  cornerAccent: ">" | "+";
  layout: Layout;
  contentType: ContentType;
  headlineSize: HeadlineSize;
  bodyWeight: BodyWeight;
  textAlign: TextAlign;
  cornerSize: CornerSize;
  accentBar: AccentBar;
  ghostAccent: GhostAccent;
}

interface ComplianceCheck {
  pass: boolean;
  detail: string;          // human-readable explanation when failing
}

interface ComplianceResult {
  colorValid: ComplianceCheck;   // accentColor matches division canonical
  headlineOk: ComplianceCheck;   // > 0 and <= 80 chars
  bodyOk: ComplianceCheck;       // > 0 and <= body max (content-type-aware)
  sourcePresent: ComplianceCheck;
  pass: boolean;
}

const ACCENT_COLORS = { Analysis: "#fe6203", Projects: "#2c40e8", Culture: "#fe43a7" };

interface ChatMessage {
  role: "user" | "assistant";
  content: string;
}
```

## Auth flow

1. User visits any page -> `proxy.ts` checks `bainsa-auth` cookie
2. No valid cookie -> redirect to `/login`
3. Submit password -> Server Action compares against `DASHBOARD_PASSWORD` env var
4. Match -> sets `bainsa-auth` cookie (HMAC token, 7-day expiry) -> redirect to today's stories
5. Mismatch -> redirect to `/login?error=1`
6. Rate limit: 5 failed attempts -> 15min lockout (in-memory)

## API security

All mutation endpoints require `X-Requested-With: fetch` header (CSRF prevention). Date params validated with `isValidDate()` regex (path traversal prevention).

## Cross-component communication

- **`stories-changed` CustomEvent**: Dispatched by AgentChat, StoryChat (on background auto-save), and StoryGrid's regenerate after any agent reply. StoryGrid listens and refetches stories + stale status.
- **Trackers + `useSyncExternalStore`** (`agentTracker.ts` factory): Four module-level trackers keep loading/unread/changed state alive across unmount and date navigation â€” `storyChatTracker`, `agentChatTracker`, `variantsTracker`, `regenerateTracker`. Each request captures its context (date, agent, sessionId) in closures so it completes cleanly even if the component unmounts; on unmount-mid-request, StoryChat auto-saves via the update API and sets the tracker's "changed" flag. See individual tracker files for the exact marker/key semantics â€” don't duplicate them here.
- **`editing` state update**: StoryGrid updates `editing` when stories refresh, so StoryEditor receives background changes via prop. StoryEditor detects prop changes and shows the diff banner.
- **`onSaved` callback**: StoryEditor calls this to update StoryGrid's in-memory stories after a save.
